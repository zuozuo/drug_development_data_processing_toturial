## 1. 什么是 aCRF

### 1.1 基本概念

**aCRF（annotated Case Report Form）** 是在空白病例报告表（blank CRF）上添加 SDTM 变量注释的文档。

> **小白类比**：
> - **CRF** = 医生在临床试验中填写的"问卷"
> - **aCRF** = 在这份问卷上标注"每个问题对应数据库里的哪个字段"

### 1.2 名称解释

```
annotated Case Report Form
    │         │      │
    │         │      └── Form：表格/表单
    │         └── Report：报告
    │
    └── annotated：带注释的 / 标注的

= 带注释的病例报告表
```

### 1.3 CRF vs aCRF

```
对比图
┌─────────────────────────────────┐
│        CRF（空白表）            │
├─────────────────────────────────┤
│ 患者姓名缩写：[    ]            │
│ 出生日期：[    /    /    ]      │
│ 性别：□ 男  □ 女                │
│ 身高(cm)：[      ]              │
└─────────────────────────────────┘

        ↓ 添加 SDTM 注释 ↓

┌─────────────────────────────────┐
│       aCRF（带注释表）          │
├─────────────────────────────────┤
│ 患者姓名缩写：[    ] ←DM.SUBJID │
│ 出生日期：[    /    /    ]      │
│             ↑ DM.BRTHDTC        │
│ 性别：□ 男  □ 女 ← DM.SEX      │
│ 身高(cm)：[      ] ← VS.VSORRES │
│                    (VSTESTCD=HT)│
└─────────────────────────────────┘
```

## 2. 为什么需要 aCRF

### 2.1 FDA 审评的需求

FDA 审评员需要理解：**数据库里的数字是从哪里来的？**

```
审评场景示例：

审评员看到 SDTM 数据：
DM.xpt
┌──────────┬────────┬────────┬─────┐
│ SUBJID   │ BRTHDTC│  SEX   │ ... │
├──────────┼────────┼────────┼─────┤
│ 001      │1985-03-│   M    │ ... │
│ 002      │1990-07-│   F    │ ... │
└──────────┴────────┴────────┴─────┘

审评员的问题：
"SEX 这个变量，是医生根据什么填写的？
 是问患者的？还是看身份证的？
 在 CRF 上怎么采集的？"

aCRF 告诉审评员：
"请看 aCRF 第3页，Demographics 部分，
 SEX 对应的采集项是'性别'单选框"
```

### 2.2 aCRF 的核心价值

```
aCRF 的价值
│
├── 🔍 可追溯性（Traceability）
│   └── 建立：CRF 采集项 → SDTM 变量 的映射
│
├── 📖 透明性（Transparency）
│   └── 审评员可以验证数据来源
│
├── 🛡️ 合规性（Compliance）
│   └── 满足 FDA 技术拒收标准要求
│
└── 📚 文档化（Documentation）
    └── 作为数据标准化过程的证据
```

### 2.3 必须提供 aCRF 的原因

```
⚠️ FDA 技术拒收标准（TRC）明确要求：
├── 所有研究数据提交必须包含 aCRF
├── 缺少 aCRF = 提交不完整
└── 可能导致提交被拒收或信息请求（IR）
```

## 3. aCRF 的组成要素

### 3.1 基础结构

```
aCRF 文档结构
│
├── 📄 封面/首页
│   ├── 研究编号（Study ID）
│   ├── 研究标题
│   └── 版本/日期
│
├── 📋 CRF 页面（按访视顺序）
│   ├── 筛选访视（Screening）
│   ├── 基线访视（Baseline）
│   ├── 治疗期访视（Treatment）
│   └── 随访/结束访视（Follow-up/End）
│
├── 🏷️ SDTM 注释
│   └── 在每个采集项旁标注对应的 SDTM 变量
│
└── 📑 注释图例/说明
    └── 解释注释的格式和约定
```

### 3.2 注释内容

| 注释类型 | 内容 | 示例 |
|---------|------|------|
| **域名** | SDTM Domain | DM, VS, AE |
| **变量名** | Variable Name | SUBJID, VSTEST |
| **测试代码** | Test Code（如适用） | VSTESTCD=SYSBP |
| **代码表** | Controlled Terminology | SEX: M=Male, F=Female |
| **补充变量** | SUPP-- 变量 | SUPPDM.QVAL |

## 5. aCRF 与数据追溯性

### 5.1 追溯链条

```
完整的数据追溯链
│
│  📝 原始记录（Source Document）
│     病历、检验报告等
│        │
│        ▼
│  📋 CRF 采集
│     临床试验专用表格
│        │
│        ▼
│  📄 aCRF 映射 ←── 我们在这里建立联系
│     CRF → SDTM
│        │
│        ▼
│  💾 SDTM 数据
│     标准化原始数据
│        │
│        ▼
│  📊 ADaM 数据
│     分析就绪数据
│        │
│        ▼
│  📈 TLFs
│     统计表格/图形
│        │
│        ▼
│  📖 CSR
│     临床研究报告
│
```

### 5.2 aCRF 在追溯链中的作用

```
追溯性示例：验证一个安全性发现

CSR 报告："3% 的患者出现头痛"
    │
    ▼
TLF 表格：AE Summary Table 显示头痛 15 例
    │
    ▼
ADaM (ADAE)：筛选 AEDECOD="Headache" 确认 15 条记录
    │
    ▼
SDTM (AE)：原始 AE 数据中 AETERM 包含"头痛"相关术语
    │
    ▼
aCRF：确认 AETERM 来自 CRF 第 X 页的"不良事件名称"字段
    │
    ▼
CRF：查看医生实际填写的不良事件报告
    │
    ▼
源文档：病历记录中患者主诉"头痛"

✓ 追溯链完整，数据可信
```

## 6. 制作 aCRF 的流程

### 6.1 制作步骤

```
aCRF 制作流程
│
├── Step 1：收集空白 CRF
│   ├── 获取最终版本的 CRF
│   ├── 确保包含所有访视和页面
│   └── 注意 CRF 版本控制
│
├── Step 2：创建 CRF-SDTM 映射表
│   ├── 列出所有 CRF 采集项
│   ├── 对应 SDTM 域和变量
│   └── 记录特殊处理逻辑
│
├── Step 3：在 CRF 上添加注释
│   ├── 使用 PDF 注释工具（如 Adobe Acrobat）
│   ├── 遵循一致的注释格式
│   └── 使用清晰的颜色/样式区分
│
├── Step 4：审核验证
│   ├── 检查注释完整性
│   ├── 验证与 Define.xml 一致性
│   └── 同行评审
│
└── Step 5：最终输出
    ├── 导出为 PDF 格式
    ├── 确保注释可搜索
    └── 文件命名：acrf.pdf
```

### 6.2 注释工具

| 工具类型 | 示例 | 特点 |
|---------|------|------|
| **PDF 编辑器** | Adobe Acrobat | 功能全面，广泛使用 |
| **专业工具** | Formedix, Pinnacle 21 | 自动化程度高 |
| **免费工具** | PDF-XChange, Foxit | 基本功能满足需求 |

### 6.3 注释样式建议

```
推荐的注释样式
│
├── 颜色：蓝色或红色（与 CRF 原文区分）
├── 字体：等宽字体（如 Courier）便于对齐
├── 格式：域名.变量名（如 DM.SUBJID）
├── 位置：紧邻对应的 CRF 字段
└── 箭头：用箭头或线条连接采集项和注释
```

## 7. 常见问题与最佳实践

### 7.1 常见问题

#### Q1: 所有 CRF 页面都需要注释吗？

```
✅ 需要注释的页面：
├── 所有采集临床数据的页面
├── 包括电子 CRF（eCRF）截图
└── 所有访视的相关页面

❌ 不需要注释的页面：
├── 纯说明性页面（指导语）
├── 签名页（如无数据采集）
└── 封面/目录
```

#### Q2: 电子 CRF（eCRF）怎么处理？

```
eCRF 的 aCRF 处理方式：
├── 截取 eCRF 界面截图
├── 将截图整理成 PDF
├── 在截图上添加 SDTM 注释
└── 最终输出为 acrf.pdf
```

#### Q3: CRF 有多个版本怎么办？

```
多版本 CRF 处理：
├── 方案一：合并所有版本到一个 aCRF
│   └── 标注每页适用的 CRF 版本
│
└── 方案二：提供主版本 aCRF
    └── 附带版本差异说明文档
```

### 7.2 最佳实践

```
✅ aCRF 最佳实践清单
├── 1. 注释完整：每个采集项都有对应注释
├── 2. 格式一致：整个文档使用统一的注释格式
├── 3. 清晰可读：注释不遮挡 CRF 原始内容
├── 4. 与 Define.xml 一致：变量名拼写完全匹配
├── 5. 包含代码表：重要的受控术语列出可选值
├── 6. 标注派生：派生变量注明"Derived"
├── 7. 版本控制：文档包含版本号和日期
└── 8. PDF 可搜索：确保注释文本可搜索
```

## 8. 合规要点

### 8.1 FDA 技术拒收标准（TRC）

```
⚠️ aCRF 相关的技术拒收标准
├── 缺少 aCRF 文件
├── aCRF 不完整（缺少重要页面）
├── aCRF 与实际数据不匹配
└── aCRF 格式不正确（如无法打开）
```

### 8.2 提交前检查清单

在提交前，确认以下内容：

- [ ] aCRF 包含所有 CRF 页面
- [ ] 每个数据采集项都有 SDTM 注释
- [ ] 注释格式一致且清晰
- [ ] 变量名与 Define.xml 完全一致
- [ ] 受控术语与 CDISC CT 一致
- [ ] 文件格式为 PDF
- [ ] 文件命名为 acrf.pdf
- [ ] 文件大小合理（建议 < 50MB）
- [ ] PDF 可正常打开和搜索

### 8.3 常见错误

| 错误 | 影响 | 避免方法 |
|-----|------|---------|
| 变量名拼写错误 | 追溯链断裂 | 与 Define.xml 交叉核对 |
| 遗漏 CRF 页面 | 注释不完整 | 使用 CRF 页面清单核对 |
| 代码表不一致 | 数据对应错误 | 使用 CDISC CT 标准术语 |
| 版本不匹配 | 审评困惑 | 确保使用最终版 CRF |
