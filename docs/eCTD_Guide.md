## 1. 一个故事：为什么需要 eCTD？

### 过去：纸质申请的噩梦

想象一下，20 年前申请新药上市是这样的：

```
📦 纸质申请包
─────────────
一个新药申请 = 几十万页文档

• 装满 100+ 个纸箱
• 重达数吨
• 运输需要卡车
• 审评员需要在纸堆里翻找
• 跨国提交需要空运多个国家
• 每次补充材料又是几箱

审评员的痛苦：
"第 153,287 页提到的那个数据在哪？"
"让我找找...可能在第 47 箱里..."
```

### 现在：电子化的便利

```
💻 eCTD 电子提交
─────────────────
同样的内容 = 一个结构化的电子包

• 存储在硬盘/服务器上
• 只有几 GB 大小
• 通过网络瞬间传输
• 全文检索，秒级查找
• 全球统一格式，一次准备多国提交
• 增量更新，只提交变化部分

审评员的便利：
"搜索'血压'..."
"找到 1,234 处，按相关性排序显示"
```

### eCTD 诞生的原因

```
┌─────────────────────────────────────────────────────────────────┐

│                                                                 │

## 2. eCTD 一句话解释

> **eCTD = 电子通用技术文档，是向药监局提交新药申请材料的国际标准格式。**

### 名称拆解

```
eCTD = electronic Common Technical Document

e       = electronic（电子的）
         不是纸质的，是电子文件

C       = Common（通用的）
         全球通用，不是某个国家专用

T       = Technical（技术的）
         包含技术性内容（药理、毒理、临床数据等）

D       = Document（文档）
         是一套文档，不是单个文件
```

### 类比理解

| 类比 | 说明 |
|------|------|
| **简历模板** | 求职者按固定格式写简历，HR 容易筛选 |
| **税务申报表** | 纳税人按固定格式填表，税务局容易处理 |
| **建筑图纸标准** | 建筑师按标准画图，审批部门容易审核 |
| **eCTD** | 药企按标准格式提交申请，药监局容易审评 |

### 谁制定的标准？

```
ICH（国际人用药品注册技术协调会）
─────────────────────────────────

成员包括：
• 🇺🇸 美国 FDA
• 🇪🇺 欧洲 EMA
• 🇯🇵 日本 PMDA
• 🇨🇳 中国 NMPA（观察员）
• 🇨🇦 加拿大 Health Canada
• 🇨🇭 瑞士 Swissmedic
• 🇧🇷 巴西 ANVISA
• 以及制药行业协会

目标：协调各国药品注册要求，减少重复工作
```

## 5. eCTD 的生命周期

### 不是一次性提交

eCTD 不是提交一次就完事，而是有完整的生命周期：

```
eCTD 生命周期
─────────────

┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 初始提交 │────▶│ 补充提交 │────▶│ 序列更新 │────▶│ 上市后  │
│Sequence │     │Sequence │     │Sequence │     │ 变更    │
│  0000   │     │  0001   │     │  0002...│     │         │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

### 序列（Sequence）概念

```
什么是序列？
────────────

每次提交都有一个序列号（Sequence Number）

序列 0000：初始申请
├── 包含完整的申请材料
└── 这是第一次提交

序列 0001：第一次补充
├── 可能是回答审评问题
├── 可能是补充新数据
└── 只包含变化的部分！（增量更新）

序列 0002：第二次补充
└── 继续增量更新...

序列 0003, 0004, ...
└── 持续更新直到获批

序列 0005+（获批后）：
├── 年度报告
├── 安全性更新
├── 标签变更
└── 生产工艺变更
```

### 增量更新的优势

```
传统方式 vs eCTD 增量更新
─────────────────────────

传统方式：
每次补充都要提交完整材料
├── 浪费时间
├── 浪费资源
└── 容易混淆版本

eCTD 增量更新：
只提交变化的部分
├── new（新增）：添加新文件
├── replace（替换）：更新现有文件
└── delete（删除）：移除过时文件

示例：
初始提交 100GB
第一次补充 500MB（只有新增/变化的部分）
第二次补充 200MB（只有新增/变化的部分）

审评员看到的是合并后的完整视图
```

## 6. 技术规范与合规要点

### 6.1 文件格式要求

```
eCTD 文件格式规范
─────────────────

📄 文档类：
├── PDF（主要格式）
│   ├── 版本：PDF 1.4 - 1.7
│   ├── 要求：可搜索文本（非扫描件）
│   ├── 书签：必须有导航书签
│   └── 安全：不能有密码保护
│
└── Word（不推荐直接提交）
    └── 应转换为 PDF

📊 数据类：
├── XPT（SAS Transport v5）
│   ├── SDTM 数据
│   └── ADaM 数据
│
└── XML
    ├── Define.xml
    ├── STF（Study Tagging File）
    └── eCTD 骨架文件

📋 命名规则：
├── 只能用小写字母、数字、连字符
├── 不能有空格或特殊字符
├── 最大长度限制
└── 扩展名要正确
```

### 6.2 技术拒收标准（TRC）

```
FDA 技术拒收标准（Technical Rejection Criteria）
─────────────────────────────────────────────────

⚠️ 以下问题会导致提交被自动拒收：

1. ts.xpt 缺失或格式错误
   └── Trial Summary 是验证规则版本的依据

2. STF（Study Tagging Files）缺失
   └── 没有节点标签，系统无法识别
